A. All concepts implemented in Phases 0 & 1, condensed

Each concept below was applied for real somewhere in the files that follow in Part B — this is the "why," collected in one place before the "what."

Phase 0 concepts

Version control (Git)

Every change tracked as a chain of commits; a staging area lets a commit be built deliberately; a local repo and the GitHub remote are separate, connected only by an explicit origin pointer.

Secret hygiene before first commit

.gitignore was written before anything else specifically because this repository is public — a committed secret is recoverable from history forever, even after later deletion.

Python virtual environments

An isolated, project-scoped copy of the interpreter and its packages, so this project's dependencies can never collide with any other Python project on the machine.

Dependency resolution

Exact version pins (==) left no room for two packages' conflicting sub-requirements to be reconciled; minimum pins (>=) gave the solver room to find a version that satisfies everyone.

Python packages and modules

__init__.py marks a folder as importable; dot-notation imports (agent.db.repositories) walk that package structure.

Centralised configuration

Exactly one file (config.py) reads environment variables; every other module asks it for a named value, never the OS directly.

Node.js / npm / package.json

The JavaScript equivalent of the Python venv + requirements.txt pairing — an isolated runtime and a declared, reproducible dependency list.

Build tools (Vite)

Transforms JSX/ES-module source into plain JS/CSS/HTML a browser can run; instant reload in development, a fixed optimised bundle for production.

Component-based UI (React) & SPA

The interface is built from small functions returning markup (JSX); one HTML page loads once, and JavaScript swaps content in afterward without full reloads.

Utility-first CSS (Tailwind v4)

Styling composed from small utility classes in markup; v4's plugin-based integration removed the separate config files earlier versions required.

Progressive Web Apps

A manifest, icon set, and iOS-specific meta tags make a website installable and launchable full-screen from a phone's home screen.

Binary file formats (PNG)

Icons were generated with a hand-written PNG encoder — signature bytes, chunked structure (IHDR/IDAT/IEND), CRC32 checksums, zlib-compressed pixel data — with zero external image libraries.

Backend-as-a-Service

Supabase supplies a managed Postgres database, an auto-generated REST API over it, authentication, and a realtime layer, replacing a hand-built backend server.

JWTs and least-privilege API keys

Both Supabase keys are signed JSON Web Tokens declaring a role; the low-privilege anon key is safe to expose publicly only because Row Level Security constrains what it can do, while the powerful service-role key is confined to the trusted server-side agent only.

Phase 1 concepts

Relational database structure

Nine tables, each one entity, related to one another rather than duplicating data.

SQL DDL vs DML

CREATE TABLE statements define structure (DDL); the day-to-day reads and writes the agent performs are DML, generated for us by the client library.

UUID primary keys

128-bit identifiers generated independently by any process with no central coordination and negligible collision risk — chosen over auto-incrementing integers because both the agent and dashboard can create rows concurrently.

Foreign keys & referential integrity

references constraints, with explicit on delete cascade / on delete set null behaviour, so the database itself refuses to let a relationship point at nothing.

JSONB columns

Variable-shaped data (a lead's weak-points list) stored as a queryable JSON document inside one column, rather than forcing a rigid extra table for a simple list.

Indexes

Separate lookup structures built on the columns queried most often (status, temperature, platform), trading a little storage/write cost for much faster reads.

Row Level Security

Enabling RLS flips a table's default from open to closed; explicit policies then grant back exactly the access needed — here, full access for any authenticated user, none for anyone else. The service-role key bypasses this layer entirely by design.

Repository design pattern

Every database read/write funnelled through one file of small, purpose-named functions, so a schema change only ever requires editing one place.

REST over Postgres (PostgREST)

The Python client never writes raw SQL during normal operation — it builds HTTP requests against an API Supabase auto-generates directly from the schema.

Smoke testing & round-trip verification

A fast, broad check (connection, tables, seed data) plus a genuine round-trip: insert, read back, confirm an exact match, then delete — leaving no trace.

DNS, IPv4 vs IPv6, connection pooling

A hostname must resolve via DNS before it's reachable; Supabase's direct-connection host is IPv6-only, unreachable on an IPv4-only network, which is exactly what was hit — solved by switching to the IPv4-reachable session-pooler endpoint instead.

Least privilege

Every component was given only the access it needs — anon+RLS for the public dashboard, service-role for the trusted agent, and a one-time DB password used only transiently and then discarded rather than stored anywhere permanent.

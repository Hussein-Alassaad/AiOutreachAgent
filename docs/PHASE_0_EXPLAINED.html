1. What Phase 0 was for

Phase 0 builds no features. Its only job is to create a clean, running skeleton so that from Phase 1 onward every piece of real work has somewhere obvious to go. The build plan defines the goal as: "a clean, running skeleton for both the agent and the dashboard."

Concretely, the phase is complete when four things are true:

  • The folder structure matches 03_PROJECT_STRUCTURE.md exactly.
  • The Python agent runs and reports its own configuration state.
  • The React dashboard builds and displays a page.
  • Every environment variable the project will ever need is documented in one place.
All four are verified done. The agent was executed and printed its self-check; the dashboard compiled to a production build in 4.26 seconds with zero errors. Neither does real work yet — that is correct and expected at this stage.

The one thing added beyond the plan

You asked that nothing from the spec be missed. A promise is not checkable, so I created REQUIREMENTS_COVERAGE.md — a ledger giving every single requirement an ID and mapping it to the phase that delivers it. A phase cannot be closed until all of its rows are marked done. This is the mechanism that makes "nothing missed" a fact rather than a hope.

3. Repository-level files explained

.gitignore

What it does: tells git which files to never track.

Why it was the very first file written: your repository is public. Once a secret is committed and pushed, deleting it later does not remove it — it stays in the git history forever, and bots scan public GitHub for exactly this. Writing .gitignore before the first git add means an .env file can never be committed by accident.

What it blocks: all .env variants, *.pem and *.key, venv/, node_modules/, __pycache__/, dist/, Playwright browser profiles (which will contain your logged-in LinkedIn and Instagram sessions), and editor/OS clutter.

Verified: after committing, I ran a check confirming no .env file is tracked.

.env.example

What it does: documents every environment variable the project will ever need, with placeholder values instead of real secrets. It is committed; the real .env files are not.

Why it matters: it is the answer to "what do I need to fill in?" Each block carries a comment saying which phase actually requires it, so you can see that an empty Claude API key is fine now but blocks Phase 4.

Structure: two sections. The first is for agent/.env (Supabase URL and service role key, Claude key, model names, WhatsApp credentials, timezone, headless flag, a dev safety cap). The second is for dashboard/.env (Supabase URL and anon key only).

Note it contains: proxy credentials are deliberately not here. They live per-account in the Supabase accounts table so you can edit them from the dashboard — which is what makes "add proxies later" a settings change with zero code change, exactly as the spec requires.

agent/config.py

What it does: reads every secret and setting from agent/.env, and is the only file that does so.

Why one file: when something is misconfigured there is exactly one place to look. Nothing else in the codebase touches environment variables directly.

The pieces:

  • _get() — reads a variable and treats an empty string as missing. Without this, a line like ANTHROPIC_API_KEY= would count as "set" and you would get a confusing authentication failure later instead of a clear message now.
  • Supabase URL and service role key.
  • Claude API key, plus the two model names as variables so the models can be changed without editing code.
  • WhatsApp provider settings — defaulted to Twilio, but this is open question Q1.
  • TIMEZONE (default Asia/Beirut) — how per-account run times are interpreted.
  • HEADLESS — when false, Playwright shows a visible browser window. You will want that while we build discovery in Phase 3 so you can watch it work.
  • DEV_MAX_LEADS_PER_ACCOUNT — a safety brake that caps volume during local testing regardless of the dashboard setting. Protects your accounts from an accidental full-volume run during development.
  • missing_required() — each phase asks only for the keys it needs, so Phase 0 runs happily with a totally empty .env while Phase 4 can refuse to start without a Claude key.

agent/db/client.py

What it does: creates and shares the Supabase connection.

  • Lazy creation. The connection is made on first use, not on import. This is why the agent runs today with no credentials at all — importing the module cannot fail.
  • One shared client. Creating a client opens a connection pool, so the module caches a single instance rather than making a new one per call.
  • Service role key. The agent runs on a server you control and needs full access to every table. See section 7 for why this key must never appear in the dashboard.
  • check_connection() — proves the credentials actually work by selecting one row from settings. It returns (ok, message) instead of raising, because in the Phase 0 self-check a failure is information, not a crash.

5. The dashboard explained

dashboard/package.json

What it does: lists JavaScript dependencies and the commands to run.

Dependencies: React 18, React DOM, React Router (for navigating between the 10 sections in Phase 9), and the Supabase JavaScript client. Build tools: Vite 6, the React plugin, and Tailwind 4.

Commands: npm run dev (development server), npm run build (production build for Vercel), npm run preview (test the production build locally).

dashboard/vite.config.js

What it does: configures the build. Three settings, each deliberate:

  • Plugins — React plus Tailwind v4. In v4 Tailwind is a Vite plugin, which is why there is no tailwind.config.js.
  • css.postcss: {} — added mid-build to fix a real failure. See section 9, problem 3.
  • server.host: true — exposes the dev server on your local network so you can open the dashboard on your phone during development, not just the laptop. Given that Mahmoud will use this on an iPhone, testing the mobile layout early matters.

dashboard/index.html

What it does: the page shell React mounts into. Most of it is the PWA setup that makes "Add to Home Screen" work properly on Mahmoud's iPhone:

  • viewport-fit=cover — lets the app draw into the notch area.
  • apple-mobile-web-app-capable — this is the tag that makes iOS open it full-screen with no Safari address bar. Without it, "Add to Home Screen" produces a bookmark rather than something that feels like an app.
  • apple-mobile-web-app-title — the name under the home screen icon.
  • apple-touch-icon — the icon itself.
  • link rel="manifest" — the manifest below.

dashboard/public/manifest.json + icons/

What it does: declares the app's name, colours, icons, and that it should launch standalone (full-screen, no browser chrome).

The icons are 180px (iOS home screen), 192px and 512px (the manifest sizes), plus an SVG favicon. I generated them programmatically — a dark slate square with a light "N" — using a small pure-Python PNG encoder, so no image library was needed. These are placeholders. Replace them with real Nexaris branding in Phase 9.

dashboard/src/lib/supabase.js

What it does: the browser's database connection.

  • Uses the anon key — never the service role key. See section 7.
  • Exports isConfigured so the UI can show a helpful setup message instead of crashing when .env is empty, which is the state right now.
  • Enables Realtime, which is what will make the live feed, run status, and analytics update by themselves as the agent works — no refresh, no polling.

dashboard/src/index.css

What it does: imports Tailwind and sets three things.

  • Temperature colours — hot, warm, and cold defined once under @theme. The Hot/Warm/Cold badge appears on live feed cards, pipeline cards, the approval queue and client history; defining the colours in one place means every screen agrees on what "hot" looks like.
  • Safe-area padding — keeps content clear of the iPhone notch and home indicator in full-screen mode.
  • -webkit-text-size-adjust — stops iOS Safari auto-enlarging text in landscape.

dashboard/src/main.jsx and src/App.jsx

main.jsx mounts React into the page — four lines, standard.

App.jsx is the Phase 0 status page. It shows whether the dashboard built correctly and whether Supabase credentials are present, then lists the 10 sections coming in Phase 9 so the build order is visible. It is written responsively (max-w-3xl, sm: breakpoints) because every screen in this project has to work on both a laptop and a phone.

7. Key concepts

The two Supabase keys — the most important thing on this page

Supabase gives you two keys and they are not interchangeable.

KeyUsed byBehaviour
anonDashboard Safe to ship in browser code. It is restricted by Row Level Security — the rules we write in Phase 1 that say only you and Mahmoud may read or write.
service roleAgent Bypasses Row Level Security entirely. Full read/write on every table. Safe only on a server you control.
If the service role key ever ends up in the dashboard, it gets compiled into the JavaScript that every visitor downloads — and since your repository is public, anyone could take it and read or delete your entire database. This is why the two clients are separate files reading separate variables, and why .env is gitignored. Whenever you copy keys, check twice which one you are pasting.

Why a virtual environment (venv)

A venv is a private Python installation for this project only. Its packages cannot conflict with other Python work on your machine, and requirements.txt lets the server rebuild the identical environment. That is why commands use agent/venv/Scripts/python.exe rather than plain python.

What "PWA" actually buys you

A Progressive Web App is a website that can be installed. On Mahmoud's iPhone: open in Safari, Share, Add to Home Screen — it then gets an icon and opens full-screen with no address bar. No App Store, no Apple developer account, no separate iOS codebase. One React app serves your laptop and his phone.

Why the agent and dashboard are separate

They share one Supabase database but are otherwise independent. The agent writes what it discovers; the dashboard reads it and writes back settings, approvals, and manual actions. They are deployed to different places (agent to an always-on server, dashboard to Vercel) and can be restarted independently — a dashboard deploy never interrupts a run.

9. Problems hit during the build, and how they were fixed

Problem 1 — Python dependencies could not be installed

Symptom: ERROR: ResolutionImpossible.

Cause: I had pinned every package to an exact version. But supabase and anthropic each require httpx within their own version range, and my exact httpx pin satisfied neither.

Fix: changed exact pins (==) to minimum pins (>=) and left httpx unpinned so pip can pick a version that satisfies both. Documented in the file itself so it is not "fixed" back later.

Problem 2 — npm install reported success but installed nothing

Symptom: exit code 0, empty output, no node_modules, and the build then failed on a missing package.

Cause: the first run was launched in the background and did not complete properly.

Fix: re-ran it in the foreground — 92 packages installed in 56 seconds. Lesson worth keeping: "the command exited 0" is not proof the work happened; I check the actual artifact.

Problem 3 — the build broke because of an unrelated project on your Desktop

Symptom: `@layer base` is used but no matching `@tailwind base` directive is present, with a stack trace pointing at C:\Users\AUB\Desktop\node_modules\tailwindcss.

Cause: this was the interesting one. When Vite starts it searches upward through parent folders for a PostCSS configuration. Your Desktop contains a leftover package.json, postcss.config.js, tailwind.config.js and node_modules from an earlier project using Tailwind v3. Our project sits inside the Desktop folder, so Vite walked up, found that v3 config, and tried to apply v3 processing to our v4 stylesheet.

Fix: declared an empty PostCSS config inside our own vite.config.js. That stops the upward search at our project boundary. I did not touch or delete anything on your Desktop.

Worth knowing: those stray files will keep interfering with any JavaScript project you create inside your Desktop folder. Cleaning them up eventually is a good idea — but that is your call, not mine to make.

Problem 4 — garbled characters in the agent output

Symptom: the self-check printed AGENT � setup.

Cause: the Windows console uses an older character encoding that cannot display the long dash character (—).

Fix: replaced the dashes in printed output with plain hyphens. Comments and documentation keep proper typography; only what reaches the console was changed.
